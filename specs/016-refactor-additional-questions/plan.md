# Implementation Plan: Refactor Additional Questions Module

**Branch**: `016-refactor-additional-questions` | **Date**: 2026-04-17 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `/specs/016-refactor-additional-questions/spec.md`

## Summary

Decompose `AdditionalQuestions.tsx` (~930 lines, cognitive-complexity violations) into a slim orchestrator (~150 lines) plus four section components and three shared UI primitives — with zero user-observable behaviour change. All state and validation logic stays in the parent; section components are purely presentational.

**Constraints**: Zero breaking changes; existing tests must pass unmodified; no new npm dependencies  
**Scale/Scope**: 1 page component, 4 section components, 3 shared UI primitives, 1 types file

## Implementation Plan

### Phase 0: Research ✅ Complete

All unknowns resolved. Key findings:

- **WorkforcePage pattern** (`src/pages/workforce/`) is the established precedent: parent holds state, sibling files export presentational sections.
- **FieldError duplication**: the `<div className="flex items-center gap-2"><InputInfo/><span>` pattern is repeated ~8 times.
- **RadioGroup/Checkbox duplication**: same label + error + input group pattern appears in 3 of 4 sections.
- **State is all in parent**: no state needs lifting; it already lives in `AdditionalQuestions`.
- **Test is safe**: existing test imports `default` from the same path and mocks all base components. It will not need changes.
- Full findings: [research.md](./research.md)

### Phase 2: Implementation Breakdown

Implementation proceeds in 5 phases, each independently verifiable:

#### Step 1 — Shared Types (`src/types/additionalQuestionsTypes.ts`)

Create the types file. Zero risk: it adds interfaces not used until later steps.

**Verify**: `pnpm run type-check` — zero errors.

---

#### Step 2 — Shared Primitive Components

Create in order (each depends on FieldError):

1. `src/components/common/FieldError.tsx`
2. `src/components/common/QuestionRadioGroup.tsx` (uses `FieldError`)
3. `src/components/common/QuestionCheckboxGroup.tsx` (uses `FieldError`)

None of these are imported by existing files yet. Safe standalone step.

**Verify**: `pnpm run type-check` — zero errors.

---

#### Step 3 — Section Components

Create all four section files. Move data arrays. Wire props with correct types from `additionalQuestionsTypes.ts`.

1. `WorkforceSection.tsx`
2. `CompensationSection.tsx`
3. `BenefitsRetirementSection.tsx`
4. `GoalsSection.tsx`

None imported by parent yet. Safe standalone step.

**Verify**: `pnpm run type-check` — zero errors.

---

#### Step 4 — Slim the Orchestrator

This is the highest-risk step. Replace the four JSX blocks in `AdditionalQuestions.tsx` with section component calls. Update imports. Remove now-unused import statements.

Key preservation rules:

- `handleSubmit` body is **unchanged** — all validation logic stays exactly as-is
- `handleAnswerChange` body is **unchanged**
- Both `useEffect` hooks are **unchanged**
- The `handleClearFieldError` helper is **new** (extracted from inline `setFieldErrors` calls in handlers, but behaviour identical)

**Verify**: `pnpm run type-check` + `pnpm test --run` — zero TS errors, all test suites pass.

---

#### Step 5 — Barrel + Cleanup

Create `src/pages/additionalQuestions/index.ts`. Optionally create `src/data/monthOptions.ts` if month array was duplicated.

**Verify**: Final `pnpm run type-check` + `pnpm test --run` + smoke-test at `/dashboard` → Finch page.
